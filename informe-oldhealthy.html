Descripción general del proyecto

Nombre OldHealthy
Propósito Proporcionar ejercicios físicos terapéuticos y personalizados para adultos mayores (60+ años) en República Dominicana, con seguimiento de progreso y recomendaciones generadas por IA.
Público objetivo Adultos mayores de 60 años con distintos niveles de movilidad, incluyendo usuarios con limitaciones físicas, uso de bastón, andadera o silla de ruedas.
Tipo de aplicación Progressive Web App (PWA) + Single Page Application (SPA). Instalable en dispositivos Android e iOS desde el navegador.
Idioma Español (República Dominicana)
Estado actual Producción — desplegada en Vercel, funcional al 100%
Repositorio Proyecto local en c:\Users\delac\OneDrive\Desktop\Idea Validation
Autor Lisandro Mercedes
Fecha del informe 21 de abril de 2026
ℹ️

OldHealthy funciona íntegramente en el navegador, sin backend propio. Toda la lógica de negocio reside en el cliente (React), y los datos se sincronizan con Supabase (PostgreSQL en la nube) de forma opcional. La IA es invocada directamente desde el navegador usando la API de Anthropic con dangerouslyAllowBrowser: true.

Flujo de datos principal

1️⃣

Onboarding: El usuario completa el cuestionario de 15 preguntas → se guarda en localStorage.userProfile → se sincroniza con perfiles_usuario en Supabase → se notifica por EmailJS.

2️⃣

Recomendaciones IA: Home.tsx lee el perfil → llama a getAIRecommendations() en anthropic.ts → se hace una llamada a Claude Haiku → el JSON de respuesta se cachea en localStorage con TTL de 1 día.

3️⃣

Ejercicio completado: ExerciseDetail.tsx finaliza el temporizador → llama a recordExercise() y addSessionRecord() en streaks.ts → actualiza streakData y sessionHistory en localStorage → sincroniza con rachas_usuario en Supabase.

4️⃣

Informe médico: Profile.tsx → navega a MedicalReport.tsx → lee sessionHistory → llama a generateMedicalReport() → el texto markdown se renderiza a HTML → el usuario puede imprimir o guardar como PDF.

05

Funcionalidades implementadas

1. Cuestionario de onboarding (Questionnaire.tsx)

15 preguntas que perfilar al usuario al inicio de la app. Tipos de pregunta: texto libre, selección única y selección múltiple. Al completarlo, el perfil se guarda en localStorage.userProfile (JSON), se sincroniza con Supabase (perfiles_usuario) y se envía notificación por EmailJS. Si el cuestionario detecta factores de riesgo, redirige a /aviso-medico antes de continuar.

Factores de riesgo: dolor severo + constante, 2+ caídas recientes, silla de ruedas combinada con nivel de dolor alto (≥4).

2. Tutorial de onboarding (OnboardingTutorial.tsx)

Modal de 7 pasos que se muestra la primera vez que el usuario accede. Usa localStorage.onboardingDone como flag. Se puede relanzar desde Perfil → botón "Ver tutorial", que despacha window.dispatchEvent(new CustomEvent("restart-tutorial")). Animaciones de entrada/salida con AnimatePresence de Framer Motion.

3. Biblioteca de ejercicios (exercises.ts + CategoryExercises.tsx + ExerciseDetail.tsx)

40+ ejercicios organizados en 4 categorías. Cada ejercicio tiene: ID único, nombre, dificultad, nivel máximo de dolor tolerable, duración, repeticiones, series, tiempo de descanso, descripción paso a paso, importancia, beneficio diario, ID de video YouTube y etiquetas.

La función getRecommendedExercises(profile, exercises) filtra ejercicios basándose en reglas: nivel de dolor, movilidad, historial de caídas y uso de ayudas para caminar.

4. Temporizador guiado con fases (ExerciseDetail.tsx — GuidedModal)

Modal interno con temporizador circular SVG. Tres fases automáticas: Ejercicio → Descanso → Completado. Al terminar todas las series, dispara confeti (canvas-confetti) y registra la sesión. El AudioContext emite un pitido al cambiar de fase. El usuario puede avanzar fase manualmente o saltar la sesión completa.

5. Sistema de rachas (streaks.ts)

Rastrea la racha de días consecutivos ejercitándose. Almacena en localStorage.streakData: racha actual, racha máxima, fecha del último ejercicio y arreglo de 7 booleanos para el histórico semanal. Se sincroniza con Supabase en cada ejercicio completado. Muestra los 7 días como puntos de color en Home y WeeklySummary.

6. Historial de sesiones (streaks.ts)

Guarda en localStorage.sessionHistory hasta 60 días de sesiones. Cada entrada contiene: exerciseId, exerciseName, categoryId, date (ISO) y durationSecs. Este historial alimenta el Resumen Semanal, la Rutina IA y el Informe Médico.

7. Resumen semanal automático (WeeklySummary.tsx)

Agrega las sesiones de los últimos 7 días. Muestra: total de ejercicios completados, minutos totales, racha actual, distribución por categoría (gráfico), y comparativa con la semana anterior (flechas de tendencia). Navega automáticamente a esta pantalla los domingos o si han pasado 7+ días desde la última visita.

8. Perfil visual (Profile.tsx)

Lee localStorage.userProfile y muestra todos los datos del cuestionario en formato visual. Secciones: datos personales (avatar con iniciales), movilidad, salud, zonas de dolor (píldoras rosas), zonas de rigidez (píldoras moradas) y objetivos (píldoras verdes). Si no hay perfil, muestra estado vacío con CTA para comenzar el cuestionario.

9. Centros de salud (MedicalCenters.tsx)

Directorio estático de 12 centros de rehabilitación en República Dominicana. Filtros: por ciudad (Santo Domingo / Santiago / Todos) y por tipo (Público / Privado). Cada tarjeta muestra: nombre, especialización, dirección, teléfono, horario, nota y botones de acción: tel:, Google Maps y wa.me (WhatsApp).

10. Notificaciones por correo (emailService.ts)

Al completar el cuestionario, se llama a EmailJS con una tabla HTML de todas las respuestas y la fecha. El correo llega a monterodionicio09@gmail.com. Esto permite al equipo de negocio conocer qué usuarios se están registrando, sin necesidad de backend propio.

Integraciones de IA

🤖

Modelo: claude-haiku-4-5-20251001 — Claude Haiku de Anthropic. Elegido por velocidad de respuesta y costo reducido, ideal para uso en tiempo real desde el navegador. Todas las llamadas se realizan desde el cliente con dangerouslyAllowBrowser: true.

Sistema de prompts

Todos los prompts usan un system prompt que establece el rol de Claude como especialista en ejercicio geriátrico, indicando: responder en español, ser empático con adultos mayores, priorizar seguridad sobre intensidad, y devolver siempre JSON válido.

El parseJSON<T> limpia las respuestas que devuelvan markdown (````json ... ````), dejando solo el JSON puro antes de parsearlo.

Reintentos automáticos

WeeklyRoutine.tsx implementa un ciclo de hasta 3 intentos con 1200ms entre ellos, mostrando "Reintentando... (2/3)" en la UI durante el proceso.

Patrón de sincronización

La app usa localStorage como fuente de verdad primaria y Supabase como sincronización secundaria. Esto permite que la app funcione completamente offline. Cuando hay conexión, los datos se sincronizan con upsert (INSERT o UPDATE según si el registro existe). La sincronización es one-way: cliente → Supabase.

Técnicas de programación usadas

Patrones de diseño

PatrónDónde se aplica

Manejo de estado

  • Estado local (useState): Estado de cada página (loading, error, resultados IA, selecciones de filtro).
  • localStorage: Fuente de verdad persistente para perfil, rachas, sesiones y cachés de IA. Sin context API ni Redux.
  • URL params: :categoryId y :exerciseId para datos efímeros de navegación.
  • Sin estado global compartido: Cada página lee directamente de localStorage al montarse.

Manejo de errores y fallbacks

  • Todas las llamadas a la API de Claude usan try/catch y devuelven null en caso de error.
  • parseJSON<T> atrapa errores de JSON malformado y devuelve null en vez de lanzar excepción.
  • Lecturas de localStorage envueltas en try/catch por posibles errores de parsing o modo privado.
  • Los componentes que dependen de IA muestran estados de carga y error específicos en la UI.
  • Botones de "Reintentar" en WeeklyRoutine y MedicalReport para errores de red.

Optimizaciones

  • Caché agresivo de IA: Evita llamadas redundantes y reduce costos de API.
  • lazy loading de ejercicios: Datos de ejercicios importados estáticamente (evita fetch dinámico).
  • Code splitting: Vite genera chunks automáticos por ruta.
  • SVG icons: Lucide React genera SVG inline sin petición de red.
  • Animaciones por GPU: motion/react usa transform y opacity para animaciones en el compositor del navegador.
  • Service Worker: Recursos estáticos servidos desde caché tras primera visita.

Variables de entorno

VariablePropósitoDónde se configura
VITE_ANTHROPIC_API_KEY API key de Anthropic para las llamadas a Claude Haiku desde el navegador Archivo .env en desarrollo, Variables de Entorno en Vercel dashboard en producción
⚠️

La API key de Anthropic está expuesta en el bundle de producción al usar dangerouslyAllowBrowser: true. Las credenciales de Supabase (URL + anon key) y EmailJS (serviceId, templateId, publicKey) están actualmente en código. Para un entorno de producción con mayor seguridad, se recomienda mover todas las llamadas a IA y servicios externos a un backend serverless (Vercel Functions o similar).
